# Codex proposal: starter-town motion

## 1. Approach

I would build a hybrid player walk cycle: generated directional character art supplies the poses, while Godot render code owns direction selection, cadence, and idle transitions. I would also add one cheap ambient effect, procedural chimney smoke, entirely in the starter-town render assembly.

For the player art, I would add `tools/art/prompts/player_character_walk_sheet.md` and generate `tools/art/out/player_character_walk_sheet.png` through the existing `generate.sh` pipeline. The prompt would specify a fixed grid with three directional rows (down, side, up) and four columns (contact, passing, opposite contact, passing), the same traveler design as the current character, a flat background, equal cell dimensions, and no contact shadow. Left-facing movement would reuse the side row with horizontal flip. Diagonal movement would select the dominant vertical or horizontal row rather than adding more AI-generated poses. This keeps the first implementation to twelve poses and avoids an eight-direction sheet whose consistency and readability would be harder to control.

I would extend `tools/art/process_assets.py` to process the sheet as a sheet rather than crop it as one sprite. It would remove the shared flat background, divide the known grid into cells, crop each cell to a common union bounding box, normalize every frame onto an equal transparent canvas with feet aligned to a shared baseline, and emit one processed sheet for each existing tunic variant. The hue-shift logic would run across all cells using the existing appearance variant definitions. Scenes would continue to reference only `tools/art/out/processed/` output.

In `scenes/player.tscn`, I would replace `Sprite2D` with an `AnimatedSprite2D` at the same feet-relative offset and add a `SpriteFrames` resource containing `idle_down`, `idle_side`, `idle_up`, `walk_down`, `walk_side`, and `walk_up`. The idle animations would be single frames taken from the sheet. I would preserve the current collision shape, camera, and player origin, so y-sort and collision behavior remain unchanged.

In `src/render/town/player_controller_2d.gd`, `set_appearance()` would load the selected processed sheet and assign it as the animation texture source. `_physics_process()` would continue to derive velocity from input, then pass the normalized input vector to a render-only animation update method. That method would remember the last nonzero facing direction, select one of the three directional animations, flip the side animation for left movement, and play at a constant frame rate while velocity is nonzero. On release it would stop on the matching idle frame. Animation cadence would not be stored in `GameState`, read from `TownLayout`, or driven by a simulation tick. Godot's `AnimatedSprite2D` frame clock would own it entirely.

For ambient motion, I would add a small `GPUParticles2D` smoke puff effect as a render-only scene, likely `scenes/town/chimney_smoke.tscn`, using Godot's built-in particle primitives and gradients rather than adding a bitmap art asset. `src/render/town/starter_town.gd` would attach one instance at a render offset above each existing cottage sprite while `_build_buildings()` is already iterating the authored building entries. The attachment decision would use the existing `sprite_key == "cottage_facade"`; there would be no new field or edit in `src/sim/town_layout.gd`. Smoke would be subtle, low-count, and non-colliding. If GPU particles prove unreliable in the headless or compatibility renderer, I would use a tiny `AnimationPlayer` scene with three `Polygon2D` circles that rise, widen, and fade on a loop.

I would add active-path tests for direction-to-animation mapping, left/right flip, idle facing retention, appearance sheet selection, and confirmation that the town builds smoke only for the existing cottage entries. The existing boot-flow test would remain part of the regression gate. This approach is preferable to purely procedural whole-sprite bobbing because a bob communicates motion but not a walk cycle. It is preferable to animation state in the simulation because facing and frame cadence are presentation details with no gameplay consequence.

Explicitly out of scope: changes to `src/sim/`, NPC motion, schedules, moving doors, weather, sound, camera effects, eight unique directional rows, and a general animation framework. I would also leave the current static player asset in the pipeline until the new sheet is accepted, both as provenance and as an easy fallback.

## 2. Risks

The largest risk is AI sheet discipline. The generator may change costume details between cells, violate the grid, include shadows that jitter, or produce poses whose feet cannot be aligned cleanly. This could make a generated sheet look worse than a restrained procedural animation. My first hour would be spent generating and processing one candidate sheet, then viewing it in motion at game scale. If the grid is unusable after one prompt revision, I would reduce the prompt to three separate four-frame strips, one per direction, rather than add brittle computer-vision logic to recover arbitrary cells.

The existing tunic hue mask may recolor unintended pixels differently across newly generated frames. It will need visual checks on all twelve frames and may require a prompt-level palette adjustment or a narrower mask. Also, changing from one texture to a sheet can subtly shift the sprite's footprint, y-sort baseline, or perceived collision position. Shared canvas normalization and a feet-baseline assertion in the processor reduce that risk, but an in-game pass around building edges is still necessary.

`AnimatedSprite2D` resources are awkward if each appearance is a different sheet and the frames are declared entirely in `.tscn`. A small render-side helper may be needed to build `SpriteFrames` from atlas regions at runtime. That is acceptable, but the API should stay local to the player instead of becoming a premature general animation system.

Particle randomness is visual and not a placement decision, but it can make screenshot-based tests unstable and may render differently across Godot backends. Tests should assert node construction and configuration, not pixels. The `AnimationPlayer` fallback is more predictable if particles cause renderer trouble. Attaching smoke by `sprite_key` also couples a visual effect to a current render asset name. That is an intentional local shortcut for this milestone, not a scalable ambient-prop data model.

This proposal does not touch a protected path. In particular, it reads the existing cottage entries through the current render assembly and does not edit `src/sim/town_layout.gd` or add ambient-motion fields to authored layout data. If review rejects the `sprite_key` shortcut and requires authored smoke anchors, that would expand the work into protected `src/sim/` data and require the protected-path decision process before implementation.

## 3. Division-of-labor claim

I am best suited to own the Godot integration and verification slice: converting the current `Sprite2D` player to directional `AnimatedSprite2D` behavior, preserving its collision and feet-based y-sort contract, wiring appearance variants, and writing headless tests. This is a compact code-and-test task where I can inspect scene resources, make mechanical GDScript changes, and run the repository gate quickly.

The other resident may be better suited to own art-direction judgment if it has stronger continuity with the existing asset prompts or prior visual review context. In particular, choosing whether generated pose consistency is acceptable should not be decided from processor success alone. The art processor extension is a useful seam for joint review because it connects those concerns without requiring either resident to own the entire feature.

## 4. Rough estimate

I estimate one to two working days: roughly two to four hours for prompt iteration and sheet processing, three to five hours for player scene and controller integration, one to two hours for smoke, and two to three hours for tests and visual tuning. The estimate grows to three or four days if image generation cannot produce a coherent grid and each direction must be generated and normalized separately. Unique diagonal animation, NPC animation, or new authored ambient placement data would also materially expand the scope and is not included.
