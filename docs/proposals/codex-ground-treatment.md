# Codex proposal: authored district ground plate

## 1. Approach

I would replace visible cell rendering with one frozen, district-sized painterly ground plate. The grid remains gameplay data, but it stops being the unit of paint. This is a method change, unlike improving the two existing diamond textures, because no cell boundary survives into the final image.

The implementation would have these parts:

- Add `tools/art/build_village_ground_guide.gd`, a headless offline authoring tool. It loads `TownLayout.build_inn_green_district()` and projects the semantic `GRASS` and `PATH` cells into plate space using `src/render/iso/projection.gd`. It produces an authoring guide, not a shipped runtime texture. For every connected PATH row or column it extracts a centerline, applies bounded lateral offsets from a stable integer hash of `(ground_seed, axis, coordinate)`, rounds corners, widens locally, and rasterizes a soft grayscale lane mask. The result follows all PATH cells but wanders inside them, has an irregular junction, and loses the straight diamond-band silhouette. Hash sampling is position-based, so traversal order cannot affect it.
- Add a source recipe under `tools/art/manifests/village-ground.json` containing the fixed seed, projected bounds, plate dimensions, lane width range, feather radius, and the semantic fingerprint. The fingerprint is a hash of width, height, and the row-major `GRASS`/`PATH` values. The tool emits the fingerprint beside its guide so a test can reject stale paint after a semantic layout change.
- Author `assets/village/ground_inn_green.png` from that guide at about 2048 by 1152 pixels. It is one opaque RGBA plate with continuous mottled grass, softly worn dirt, grass incursions, thin secondary wear, and no props or object shadows. I would first use a zero-credit image generation pass conditioned on a ground-only crop of `docs/art/iso-five-asset-spike.png`, then composite and hand-correct it against the deterministic lane mask. The guide constrains topology, while the generated paint supplies texture. The final PNG is curated and frozen, so runtime determinism does not depend on reproducing a model call. I would not use Meshy for this pass because a 2D ground plate does not benefit from its 3D pipeline or paid credit risk.
- Add a `ground` record to `assets/village/manifest.json` with the plate PNG, its projected origin anchor, native size, and semantic fingerprint. `src/render/town/village_render.gd::_build_ground()` loads that record through `_load_texture()`, creates one `Sprite2D` at the declared projected origin, and does not create per-cell polygons. The plate lives under `res://assets/village/`, so `all_resources` includes it and the existing isolated-PCK audit exercises it through `ResourceLoader`. There is no `Image.load`, runtime source-PNG access, export glob change, or sim dependency on a texture.
- Extend `test/active_path/test_village_render.gd` to assert exactly one manifest-driven ground plate, no legacy ground diamonds, correct declared dimensions, and equality between the manifest fingerprint and the current semantic grid. Extend the capture gate with a district screenshot comparison at the existing 0.5x, 1x, and 2x scales. The export audit already checks every manifest texture, so the new record becomes part of the packaged-export gate automatically.
- Add soft contact shadows in `src/render/town/village_render.gd` as render-owned siblings of object sprites. A helper creates a low-alpha, feathered oval or footprint polygon from each placement footprint and kind, positions it at the same projected contact point, and inserts it in a dedicated `ContactShadowLayer` above the ground plate but below all world sprites. Large structures use broad foundation shadows, trees and props use tighter ellipses, and crown placements get no independent shadow. The shadow parameters live in render code or manifest metadata, never in `town_layout.gd`. If Godot polygon antialiasing is insufficient in capture, I would ship a small neutral blurred shadow texture through the same manifest and scale it per footprint rather than generating image pixels at runtime.

Data crosses the boundary only through the existing `TownLayout`: dimensions, semantic ground grid, and placements. The offline authoring tool reads those semantics to make a lane guide, and the runtime renderer reads the same layout only to validate the plate fingerprint and position render-owned paint and shadows. `src/sim/town_layout.gd` needs no texture, viewport, projection, mask, seed, or wander-offset field. Nav cost continues to use the unchanged PATH grid.

I would leave out transition tiles, runtime shaders, richer sim trail data, object re-slicing, and denser placement changes. The plate can visually add small wear branches, but the main dirt lane must remain inside or immediately adjacent to semantic PATH cells so the art does not advertise a traversability rule that the sim does not have. For a later 12 to 16 structure village, I would author several district plates on a shared world-space canvas or split one large authored surface into overlap-padded render chunks after painting. I would not return to independently visible tiles.

## 2. Risks

The largest risk is semantic drift. A beautiful frozen plate can lie after someone edits the PATH grid. The fingerprint makes that failure loud, but regeneration and repainting are still manual work. A second risk is that an image model may invent props, hard edges, lighting, or paths outside the guide. The output needs a ground-only prompt, masked compositing, and human correction rather than direct acceptance.

One 2048 by 1152 RGBA texture is roughly 9 MiB uncompressed before mipmaps and import overhead. That is modest for this district but does not scale indefinitely. A full village could require chunking with generous overlap and one final source canvas to prevent seams. Chunking too early would reintroduce the seam problem this proposal is meant to remove.

The plate is specific to this authored district and gives up cheap runtime rearrangement. That is consistent with the frozen authored-map pivot, but worse than blended tiles if frequent layout iteration is expected. It also makes camera exposure beyond the painted bounds obvious, so the plate needs bleed outside the 16 by 14 cell footprint and the camera limits must respect it.

Procedural `Polygon2D` shadows may still look synthetic against painterly assets, especially around irregular building foundations. A shared blurred texture improves softness but can look repetitive. Per-object shadow masks would look best but add authoring work and manifest entries. Shadows must also remain below every object while following the same contact point, or depth sorting will produce visible overlaps.

The one-hour question I would answer first is whether a 2048 by 1152 masked image-generation pass can preserve the guide's lane topology and match the spike's grass and dirt texture without adding scenery. I would build the guide for the current cross, generate one candidate, composite the mask, place the existing inn and tree over it, and capture at 1x. If that fails, I would keep the plate method but paint the surface by compositing enlarged spike-ground crops, clone texture, and hand-authored masks instead of spending calls tuning a model.

## 3. Division-of-labor claim

I am best suited to own the deterministic guide generator, semantic fingerprint, runtime plate loader, contact-shadow layer, and headless/export tests. Those pieces benefit from careful code-path and packaged-resource validation, which fit the Codex harness well.

The resident with the strongest multimodal art iteration loop should own the masked generation, compositing, and spike-confusability captures. That work depends on visual judgment across repeated 0.5x, 1x, and 2x comparisons more than on code construction. I would give that resident the exact guide, bounds, and acceptance captures rather than ask them to infer topology from code.

## 4. Rough estimate

This is about one focused worker-day for the generator, fingerprint, renderer, shadows, and tests, plus half to one art-day for two or three plate candidates and cleanup. Expect roughly 250 to 450 lines of GDScript and test changes, one manifest recipe, one runtime PNG, and one or two QA captures.

It grows to several days if the model cannot obey the lane mask, if painterly contact shadows must be authored per object, if the camera must roam well beyond current district bounds, or if this sub-round must solve chunking for the future full village instead of proving one seamless district plate now.
